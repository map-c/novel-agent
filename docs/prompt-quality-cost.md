# 提示词工程、生成质量控制与成本优化

## 一、提示词工程

### 1.1 分阶段提示词策略

不同阶段需要完全不同的提示词风格：

| 阶段 | 输出类型 | 提示词重点 |
|---|---|---|
| 输入分析 / 世界观 / 角色 / 大纲 | 结构化数据（JSON） | 约束格式、枚举选项、schema 引导 |
| 章节生成 | 自由文本 | 文学风格引导、情感基调、节奏控制 |
| 一致性检查 | 分析报告 | 比对指令、checklist 式检查 |

结构化阶段和创作阶段的提示词设计哲学相反——前者要**收敛**（精确、可控），后者要**发散**（创造力、文学性）。

### 1.2 章节生成提示词（核心难点）

#### a) 风格一致性

每一章独立调用 LLM 时，风格会漂移。应对策略：

- **风格锚点**：在提示词中注入第一章开头 300-500 字作为"文风参考"，让后续章节模仿
- **风格描述词**：提取并固定风格标签（如"冷峻克制、短句为主、多用白描、少用比喻"），每章都注入
- **负面约束**：明确说"不要"什么（"不要使用网络用语"、"不要突然切换为第一人称"）

#### b) 角色"说话方式"一致性

角色对话是最容易穿帮的地方。每个角色卡需要包含：

- 口头禅、语气词偏好
- 说话长度倾向（话多还是话少）
- 用词习惯（文雅/粗犷/书生气）
- 示例对话（2-3 句 few-shot）

#### c) 章节长度控制

LLM 生成长度很不稳定。几种策略：

- **直接要求字数范围**（"本章目标 2000-2500 字"）—— 效果一般，LLM 对中文字数感知差
- **分段生成**：一章拆成 3-4 个场景，每个场景独立生成 —— 更可控但衔接成本高
- **过长截断 + 过短续写**：生成后检测字数，不够就追加"请继续本章内容"

#### d) 情节引导的松紧度

大纲给的细节越多，生成越可控但越死板。需要找平衡点：

- 大纲只给"本章发生什么"（宏观），不规定"怎么写"（微观）
- 关键情节点必须命中，但中间的过渡、对话内容留给模型自由发挥
- 在提示词中区分 **must-have**（必须出现的事件）和 **nice-to-have**（可选的细节）

## 二、生成质量控制

### 2.1 质量问题分类

| 问题类型 | 示例 | 检测难度 | 影响程度 |
|---|---|---|---|
| 事实矛盾 | 第2章说主角是独生子，第7章突然出现妹妹 | 高 | 致命 |
| 角色名混乱 | 同一角色前后名字不一致、或张冠李戴 | 中 | 严重 |
| 情节悬空 | 第3章埋了伏笔，后续从未回收 | 高 | 中等 |
| 风格漂移 | 前几章冷峻写实，后面突然变得轻浮 | 中 | 中等 |
| 节奏失衡 | 某章全是对话没有推进，某章信息量过大 | 中 | 中等 |
| 重复表达 | 反复使用相同的描写句式或比喻 | 低 | 轻微 |

### 2.2 三层质量控制体系

#### 第一层：预防（生成前）

- 完善的上下文注入（ContextManager 的滑动窗口 + 层次化摘要）
- 明确的章节大纲约束（must-have 事件列表）
- 角色卡中包含足够的区分信息（外貌、语言风格、行为习惯）

#### 第二层：检测（生成后）

**LLM 检测（语义层面）：**

每章生成后，用另一个 LLM 调用做一致性校验：

- 检查角色名是否正确
- 检查是否与大纲偏离
- 检查是否引入了新的未定义角色/设定
- 检查情感基调是否与预期一致

**规则检测（程序化）：**

非 LLM 的低成本检查：

- 正则匹配角色名出现频率和拼写
- 字数统计（是否在目标范围内）
- 对话占比分析（避免全是对话或全是叙述）
- 重复短语检测

#### 第三层：修复（检测到问题后）

根据问题严重程度选择不同策略：

| 严重度 | 示例 | 处理方式 |
|---|---|---|
| 轻微 | 名字错别字、重复表达 | 自动修复，修复后通知用户 |
| 中度 | 情节偏离大纲 | 提示用户，提供修复建议，用户选择 |
| 严重 | 逻辑矛盾 | 建议整章重新生成 |

**修复成本提示：** 修复一章可能需要额外 1-2 次 LLM 调用。如果质量控制太激进，每章都触发修复，成本翻倍。需要设定合理的阈值。

## 三、成本优化

### 3.1 基础成本估算

以一部 2 万字中篇小说（约 10 章）为例：

| 阶段 | 输入 tokens | 输出 tokens | 调用次数 | 小计 tokens |
|---|---|---|---|---|
| 输入分析 | ~500 | ~800 | 1 | ~1,300 |
| 世界观构建 | ~800 | ~1,500 | 1 | ~2,300 |
| 角色设计 | ~1,000 | ~3,000 | 1 | ~4,000 |
| 大纲生成 | ~2,000 | ~3,000 | 1 | ~5,000 |
| 章节生成 (×10) | ~2,500 | ~3,000 | 10 | ~55,000 |
| 章节摘要 (×10) | ~3,500 | ~500 | 10 | ~40,000 |
| 一致性检查 | ~10,000 | ~1,000 | 1 | ~11,000 |
| **合计** | | | **~25次** | **~120,000** |

> 章节生成和章节摘要占了约 80% 的 token 消耗。

### 3.2 优化策略

#### a) 分级模型策略（最有效，可降低 40-60% 成本）

不是每个阶段都需要最强的模型：

| 阶段 | 推荐模型 | 理由 |
|---|---|---|
| 输入分析 | Haiku / GPT-4o-mini | 简单提取任务 |
| 世界观 / 角色 / 大纲 | Sonnet | 需要创造力但不需要最高质量 |
| **章节生成** | **Opus / GPT-4o** | 核心产出，直接决定最终质量 |
| 章节摘要 | Haiku / GPT-4o-mini | 压缩摘要是简单任务 |
| 一致性检查 | Sonnet | 需要理解力但不需要创作 |

#### b) 摘要压缩优化

章节摘要是每章都要做的高频操作：

- 用小模型（Haiku）做摘要而非 Opus
- **滚动摘要**而非累积摘要——每次更新总摘要，而不是把所有历史拼接让模型重新总结
- 如果章节内容没有重大情节推进，可以复用上一章的总摘要，只追加 delta

#### c) 缓存策略

- 世界观、角色卡、大纲这些"静态上下文"在整个生成过程中不变，可以利用 **Anthropic Prompt Caching**（固定前缀缓存，后续调用只计增量 token）
- 如果用户重新生成某一章，之前章节的上下文已在缓存中

#### d) 提示词精简

- 上下文注入的每个部分严格控制长度上限
- 角色卡只注入当前章节"出场角色"，不注入全部角色
- 避免在提示词中放冗长的格式说明，用简洁的 XML 标签替代

#### e) 失败重试策略

- 设置最大重试次数（如 2 次），超过就保留当前版本让用户人工介入
- 避免"质量不够就重新生成"的无限循环

## 四、三者的交叉影响

提示词质量、生成质量、成本三者相互制约：

```
提示词质量 ↑  →  生成质量 ↑  →  需要的修复/重试 ↓  →  成本 ↓
提示词长度 ↑  →  生成质量 ↑  →  但每次调用 token ↑  →  成本 ↑
模型降级   →  成本 ↓        →  但生成质量可能 ↓    →  修复次数 ↑
```

**核心原则：用最好的提示词 + 精准的上下文注入，在章节生成这个核心环节用强模型，其他环节用便宜模型。把预算集中在真正影响用户体验的地方。**
