# 用户交互设计

## 一、用户心智模型

不同的心智模型导致不同的交互设计：

| 心智模型 | 用户预期 | 交互风格 |
|---|---|---|
| "AI 写手" | 我给指令，它帮我写 | 命令式，用户主导 |
| "协作伙伴" | 我们一起创作 | 对话式，双向交流 |
| "自动机器" | 我按个按钮，等结果出来 | 向导式，最少操作 |

推荐采用**"协作伙伴"**模型——用户提供创意方向，Agent 做繁重的文字工作，关键决策由用户把关。

核心挑战：**如何让"协作"感觉自然，而不是在填表？**

---

## 二、输入阶段

### 2.1 自由输入 vs 引导输入

| 方式 | 优势 | 劣势 |
|---|---|---|
| 纯自由输入 | 门槛低，不限制创意 | 用户可能写得太模糊，导致生成质量差 |
| 结构化表单 | 信息完整 | 体验像填问卷，扼杀创意 |
| 渐进式引导 | 最自然，兼顾信息完整性 | 多一轮交互 |

### 2.2 推荐方案：渐进式引导

```
用户输入一段自由文字（或贴入参考简介）
        ↓
Agent 分析，提取已有信息，识别缺失的关键要素
        ↓
以对话方式追问 1-3 个关键问题（只问最影响方向的）
  例："你描述了一个民国时期的悬疑故事。我想确认几点：
       1. 主角的职业身份是什么？
       2. 悬疑的核心是案件侦破还是心理博弈？
       3. 你希望结局是光明的还是灰暗的？"
        ↓
用户回答（可以只回答部分，其余交给 Agent 决定）
        ↓
进入生成流程
```

用户觉得在"聊创意"，而不是在"填工单"。

### 2.3 参考简介模式

融入渐进式引导流程——用户贴入简介后，Agent 回应："我从这个故事中提取了以下灵感方向：A、B、C。你想保留哪些？还有什么想调整的？"

## 三、审阅阶段（最关键）

用户需要审阅世界观、角色、大纲。这是整个交互设计的重心。

### 3.1 核心矛盾

信息量大 vs 用户耐心有限。完整的大纲可能有 10+ 章节摘要、5+ 角色卡、世界观设定文档，一次性展示会导致信息过载。

### 3.2 分层展示 + 聚焦操作

```
第一层：概览（一屏可见）
├── 故事一句话概括
├── 主要角色（只显示名字和角色定位）
├── 章节列表（只显示标题）
└── 整体基调标签

第二层：详情（点击展开）
├── 角色卡完整内容
├── 各章节详细摘要
└── 世界观完整设定

第三层：编辑（在详情基础上修改）
├── 直接编辑文本
├── 或者用自然语言指令（"把这个角色改得更阴沉一些"）
```

### 3.3 编辑方式：混合模式

| 方式 | 适用场景 | 优势 | 劣势 |
|---|---|---|---|
| 直接编辑 | 改名字、删角色、调整简单描述 | 精确 | 用户需要自己写，负担重 |
| 自然语言指令 | 调整情节走向、改角色性格、调整节奏 | 轻松 | 结果可能需要多轮调整 |

推荐界面上两种方式并存，用户自由选择。

### 3.4 审阅确认模式

- 每个部分（世界观、角色、大纲）**独立确认**，不使用"全部确认"按钮
- 已确认的部分标为 ✓，可以随时回退修改
- 全部确认后才进入章节生成
- 提供**"自动确认"选项**给不想细看的用户——跳过审阅直接生成

## 四、生成阶段

逐章生成是最长的过程（可能需要几分钟到几十分钟）。

### 4.1 实时流式预览

优势：
- 给用户"正在工作"的反馈感
- 用户可以提前发现问题（比如开头就跑偏了）
- 增加参与感和期待感

问题：
- 用户盯着看会觉得慢
- 生成过程中用户能做什么？

### 4.2 界面布局

```
┌─────────────────────────────────────────────────┐
│  [章节导航]        [主内容区]        [信息面板]  │
│                                                  │
│  Ch.1 ✓           第三章：暗流                   │  世界观摘要
│  Ch.2 ✓           ─────────                      │  ──────────
│  Ch.3 ● 生成中    夜色笼罩着整座城市，            │  民国上海...
│  Ch.4 ○           霓虹灯在雨幕中晕开              │
│  Ch.5 ○           一团团模糊的光......            │  出场角色
│  ...              [文字持续流出]                  │  ──────────
│                                                  │  林探长
│                                                  │  苏小姐
│                                                  │
│  ─────────────────────────────────────────────── │
│  [进度: 3/10 章]  [⏸ 暂停]  [⏭ 跳过本章]        │
│  预计剩余: ~8分钟                                 │
└─────────────────────────────────────────────────┘
```

### 4.3 关键交互点

- **已完成的章节可以回看和编辑**，不用等全部生成完
- **暂停按钮**：生成到当前章节末尾后暂停，用户可以审阅调整再继续
- **跳过**：对某章不满意，标记为"待重写"，先继续后面的
- **右侧信息面板**：展示当前章节涉及的角色和场景，方便用户对照检查一致性

### 4.4 等待时间利用

等待时间是最容易流失用户的。应对策略：

- 允许用户回看和编辑已完成的章节（最实用）
- 显示 Agent 的"思考过程"——"正在构思第三章的开场氛围..."（增加透明感）
- 提供进度预估（基于已完成章节的平均耗时）

## 五、回退与分支

创作过程不是线性的，用户可能在第 7 章时觉得第 3 章的方向就不对。

### 5.1 需要支持的操作

| 操作 | 描述 | 影响范围 |
|---|---|---|
| 编辑已生成章节 | 直接修改文字 | 仅当前章 |
| 重新生成某章 | 保持大纲不变，重新生成正文 | 仅当前章，但后续章节的"前文摘要"需要更新 |
| 修改大纲后重新生成 | 改了大纲中某章的方向 | 该章及所有后续章节需要重新生成 |
| 修改角色设定 | 改了角色卡 | 所有涉及该角色的后续章节需要重新生成 |

### 5.2 修改后对已生成章节的处理

| 方案 | 描述 | 适用场景 |
|---|---|---|
| A. 自动重新生成 | 自动重新生成所有受影响章节 | 简单粗暴，成本高，用户等待时间长 |
| B. 标记 + 用户选择 | 标记为"可能不一致"，用户选择哪些需要重新生成 | **推荐**，灵活，用户可控 |
| C. 延迟处理 | 只更新上下文（摘要），一致性检查时再统一处理 | 最省成本，但风险较高 |

推荐**方案 B**，界面上用视觉标记（如黄色警告标）提示哪些章节可能受到影响，由用户决定是否重新生成。

## 六、完成阶段

小说生成完成后的交互需求：

- **通读全文**：提供舒适的阅读视图（非编辑视图），字体、行距适合长文阅读
- **最终编辑**：允许直接修改任何段落
- **整体润色**：提供"全文润色"功能——Agent 统一调整风格、修复小瑕疵（额外 LLM 调用）
- **导出**：Markdown / 纯文本下载

## 七、交互设计原则

1. **渐进式复杂度** —— 不在一开始就暴露所有选项，随着流程推进逐步展示
2. **双通道操作** —— 简单改动直接编辑，复杂改动自然语言指令，用户自由选择
3. **永远不要让用户干等** —— 生成过程中提供有意义的操作（回看、编辑已完成章节）
4. **可撤销而非确认弹窗** —— 不要"确定要修改吗？"，而是让操作可回退
5. **透明而非黑盒** —— 展示 Agent 当前在做什么、为什么做，而不只是一个进度条
